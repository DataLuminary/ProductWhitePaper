# DataLuminary 战略白皮书

> 面向用户、开发者与投资者的产品白皮书。本文以 DataLuminary 的完整产品形态、战略取舍与生态价值为主，具体工程迭代维护在公开 `plan/` 与 `spec/` 中。

## 执行摘要

DataLuminary 是一个开源、插件化、可私有化部署的全链路 BI 数据洞察平台。它覆盖从数据源接入、数据集建模、指标治理、图表生成、仪表盘编排、自然语言分析到 AI 洞察报告的完整流程，目标是让企业以更低成本把分散数据转化为可复用、可解释、可协作、可自动化的决策资产。

传统 BI 工具往往在两个方向上割裂：一类强调拖拽自助分析，但复杂场景仍依赖报表开发；另一类强调 AI 问数，但生成结果难以复用、治理和审计。DataLuminary 的核心判断是：下一代 BI 不会只属于拖拽，也不会只属于一句话生成，而是要同时支持三种工作方式：

- 业务用户通过拖拽和配置完成日常分析。
- 分析师与数据团队通过 AI 分步骤打磨数据集、指标、图表和仪表盘。
- 管理者以一句话描述目标，由系统自动生成可编辑、可追溯的分析草案。

这三种模式共享同一套数据集、权限、插件和 Schema，因此生成结果不是一次性截图，而是可以继续编辑、复用、发布和集成的企业数据资产。

## 市场问题

企业不缺数据，缺的是把数据持续变成行动的能力。常见痛点包括：

- 数据散落在业务系统、文件、数据库、IoT 遥测、运维日志和 Agent 运行记录中，接入链路不统一。
- BI 使用门槛高，业务人员依赖研发排期，研发团队又难以沉淀可复用的分析资产。
- 图表、大屏、移动看板、报表导出和审计视图通常被不同系统割裂，维护成本随场景增长。
- AI 问数提升了入口效率，但如果缺少数据治理、权限、指标语义和可编辑配置，难以进入企业生产流程。
- 私有化客户需要可控、安全、可扩展的部署形态，而不是只能适配单一云环境的 SaaS。

DataLuminary 的产品定位，是在开源可控的工程底座上，把 NoCode BI、插件生态、AI 数据洞察和私有化交付整合为一条连续链路。

## 产品定位

DataLuminary 面向四类核心角色：

| 角色 | 关注点 | DataLuminary 提供的价值 |
|------|--------|--------------------------|
| 业务用户 | 快速看数、做图、搭看板 | 拖拽式数据集、图表与仪表盘配置，减少报表排期依赖 |
| 数据与研发团队 | 可维护、可扩展、可治理 | 全栈 TypeScript、插件化架构、Schema 化配置与权限模型 |
| 企业 IT / 交付团队 | 私有化、安全、可适配 | PostgreSQL 默认底座、Docker 化初始化、可按数据规模扩展 |
| 投资者 / 合作方 | 长期壁垒与生态空间 | BI + AI + 插件市场 + LuminaryWorks 生态协同形成复合增长路径 |

## 全链路 BI 数据洞察

DataLuminary 的主线不是单点图表工具，而是全链路数据洞察平台：

```text
数据源接入 -> 数据集建模 -> 指标/维度治理 -> 图表生成 -> 仪表盘编排 -> AI 洞察 -> 发布与协作
```

### 1. 传统拖拽完成

对于高频、确定性的业务分析，DataLuminary 保留成熟 BI 的低门槛体验：

- 通过数据源插件接入数据库、文件、API 与业务系统。
- 在数据集层配置字段、指标、维度、过滤和预览。
- 使用图表插件生成折线图、柱状图、表格、地图、指标卡等多种视图。
- 通过 grid-layout 等面板插件拖拽组合仪表盘、大屏和看板。
- 基于空间、角色和资源权限进行协作和发布。

这条路径适合运营日报、销售漏斗、生产监控、财务分析、设备大屏等稳定场景。

### 2. AI 分步骤打磨完成

对于探索性分析，DataLuminary 的 AI 方向不是替代专业判断，而是把复杂配置拆成可验证步骤：

- 先理解业务目标，建议数据源与数据集字段。
- 再推荐指标、维度、筛选条件和聚合方式。
- 然后生成图表配置草案，并解释为什么使用该图表。
- 最后组合仪表盘，输出洞察摘要和可追溯的分析路径。

用户可以在每一步检查、修改和回退。AI 生成的是平台原生配置，而不是不可维护的黑盒结果。

### 3. 一句话自动生成

DataLuminary 的自然语言入口面向更高层级的经营问题：

> “帮我生成一个本季度销售转化漏斗，按区域对比，并标出异常波动。”

系统围绕目标自动选择数据集、生成图表组合、布局仪表盘，并给出洞察摘要。与普通聊天式问数不同，生成结果仍然落在 DataLuminary 的数据集、图表、仪表盘和权限体系内，可继续编辑、发布、审计和二次开发。

## 完整产品形态

DataLuminary 的完整形态是一套面向企业数据决策的复合平台，而不是单一仪表盘工具：

| 产品层 | 形态 | 核心价值 |
|--------|------|----------|
| 数据连接层 | 数据库、文件、API、日志、IoT、Agent 与业务系统连接器 | 把企业分散数据接入统一分析入口 |
| 数据语义层 | 数据集、指标、维度、字典、血缘、口径说明 | 把原始数据沉淀为可复用的数据资产 |
| 分析创作层 | 拖拽建模、SQL 高级模式、AI 分步骤助手、自然语言生成 | 让不同能力层级的用户都能完成分析 |
| 可视化表达层 | 图表、报表、大屏、移动看板、管理驾驶舱、嵌入式组件 | 用不同形态服务运营、管理、监控和汇报 |
| 洞察自动化层 | 异常识别、归因分析、趋势解释、预测建议、洞察报告 | 从“展示数据”升级到“解释数据” |
| 治理协作层 | 权限、空间、版本、审计、发布、订阅、告警 | 让 BI 结果进入企业生产流程 |
| 开发生态层 | 插件 SDK、Schema、模板市场、行业解决方案 | 让社区和合作伙伴持续扩展平台边界 |

这一形态下，DataLuminary 可以同时呈现为自助 BI、AI 数据分析助手、企业大屏平台、嵌入式分析引擎、行业数据应用底座和开源插件生态。

## 插件化架构与社区增长

DataLuminary 的长期技术壁垒来自插件化，而不是内置功能数量本身。平台采用微内核思路，把变化快、行业差异大的能力放到插件层：

- 数据源插件：MySQL、PostgreSQL、ClickHouse、Excel、API、IoT、日志与第三方系统。
- 图表插件：常规图表、行业图表、地图、指标卡、多媒体组件和自定义渲染。
- 面板插件：PC 网格看板、大屏自由布局、移动列表、报表卡片。
- 交互插件：筛选器、联动、下钻、时间控件、变量查询。
- AI 插件：分析建议、图表生成、报告生成、数据质量检查。

对用户而言，插件化意味着平台可以适配业务，而不是让业务迁就平台。对开发者而言，插件化意味着社区可以围绕行业、数据源、图表与交互体验持续贡献。对投资者而言，插件生态意味着产品边界可扩张，商业模式可以从软件交付延伸到模板、行业方案、插件市场和生态合作。

## 全栈 TypeScript 与 AI 一等编程语言

DataLuminary 坚持全栈 TypeScript，是因为 BI 平台同时需要复杂前端交互、后端权限与数据建模、插件 SDK、配置 Schema 和工程治理。统一语言能降低协作成本，也能让 AI 更可靠地参与开发。

我们认为 TypeScript 6.0、7.0 时代会进一步强化这一趋势：类型系统、结构化接口、工程工具链和代码可读性，会让 TypeScript 成为 AI 一等编程语言之一。DataLuminary 围绕 TypeScript 构建插件接口、配置 Schema、前后端契约与测试基线，使 AI 生成代码更容易被类型系统约束、被工程流程审查、被社区理解和维护。

这不是单纯的技术偏好，而是产品战略：当 AI 开发逐渐普及，能被 AI 稳定理解、扩展和验证的平台，会拥有更低的生态参与门槛。

## 可能演化出的产品形态

围绕同一套平台底座，DataLuminary 可以自然延展出多种产品形态：

- **AI BI Copilot**：面向业务用户的自然语言分析入口，从问题到图表、看板、解释和行动建议。
- **企业指标中台**：沉淀指标口径、维度字典、数据血缘和权限边界，支撑跨部门统一看数。
- **行业驾驶舱**：为制造、零售、教育、运维、IoT、Agent 平台等行业提供预置模板和指标体系。
- **嵌入式分析平台**：把图表、仪表盘、报表和洞察能力嵌入第三方业务系统。
- **数据应用搭建器**：通过插件、Schema 和拖拽能力构建轻量数据应用，而不只是静态看板。
- **插件与模板市场**：围绕数据源、图表、AI Agent、行业包和交付模板形成社区与商业生态。
- **私有化 AI 数据工作台**：面向高安全行业，在内网环境中提供数据分析、报告生成和审计协作。

这些形态并不互斥。它们共享 DataLuminary 的数据资产、权限治理、插件运行时和 TypeScript 工程体系，使产品具备平台化扩展空间。

## 私有化部署与规模适配

DataLuminary 面向企业场景设计，默认支持私有化部署和分层扩展：

- 轻量试点：单机或小规模 Docker 环境，快速验证数据源、图表和看板价值。
- 部门级使用：PostgreSQL 持久化、角色权限、空间隔离、稳定备份与发版流程。
- 企业级场景：按数据规模扩展查询、缓存、审计、可观测性和集成能力。
- 行业适配：通过插件接入制造、教育、运维、IoT、Agent、金融、零售等不同数据形态。

私有化的重点不是“把 SaaS 搬进客户机房”，而是让客户在数据安全、合规、网络隔离、交付周期和成本结构之间取得平衡。DataLuminary 的开源底座和 TypeScript 工程体系，使客户可以看见、验证并扩展关键链路。

## LuminaryWorks 生态协同

DataLuminary 是 LuminaryWorks 生态中的数据洞察中枢。它既能独立商业化，也能与兄弟项目协同增长：

| 生态产品 | 与 DataLuminary 的协同 |
|----------|------------------------|
| LuminaryIoTChain | 设备遥测、告警与生产数据进入 DataLuminary，形成 IoT 监控大屏 |
| VistaRemote | 远程运维、录制与审计数据进入 DataLuminary，形成效率与合规报表 |
| VibeEdu / BlockyEdu | 教学、实验和学习行为数据进入 DataLuminary，形成教育分析看板 |
| AgentSkillMesh / VibeAgent | Agent 运行、交易和技能数据进入 DataLuminary，形成 Agent 经济洞察 |
| LuminaryWorks shared / identity | 统一身份、权限契约、SDK 和跨产品开发体验 |

生态协同的核心原则是松耦合：每个社区可以独立成长、独立部署、独立商业化；当客户需要跨场景解决方案时，通过统一身份、公开契约和数据插件完成组合。

## 商业与投资逻辑

从商业视角看，DataLuminary 的机会来自三条曲线叠加：

- BI 基础需求长期存在：企业持续需要报表、大屏、监控、审计和经营分析。
- AI 改变 BI 交互方式：自然语言、自动建模、自动配图和洞察生成会降低使用门槛。
- 开源插件生态扩大覆盖面：社区贡献数据源、图表和行业模板，平台方沉淀治理、市场和企业服务能力。

因此，DataLuminary 的长期价值不只是一套 BI 功能，而是一个围绕数据洞察的可扩展平台。产品价值在于快速交付可用看板、沉淀企业数据资产，并形成 AI + BI + 插件市场 + 生态协同的增长飞轮。

## 结语

DataLuminary 的目标，是让每个组织都能用可控、开放、可扩展的方式点亮数据。它服务业务用户的即时决策，也服务开发者的长期扩展；它能作为单独 BI 产品落地，也能成为 LuminaryWorks 生态的数据洞察中枢。面向 AI 时代，DataLuminary 希望把“看见数据”升级为“理解数据、解释数据、行动数据”。
