# 自由布局大屏：固定逻辑尺寸决议

> **状态**：已决议（2026-07）  
> **受众**：产品经理、售前、设计、项目管理、AI 需求助手  
> **工程说明**：[开发者文档 · 设计与实现](../develop/dashboard-position-layout.md) · [DataView 前端规格](https://github.com/DataLuminary/DataView/blob/main/spec/development/position-layout-fixed-canvas.md)

## 1. 决议摘要

**DataLuminary 自由布局（大屏）采用「固定逻辑宽高 + 纯绝对定位 + 整体等比缩放」，不支持「固定宽度、高度随内容自由撑高」作为产品能力。**

画布逻辑尺寸为固定像素（常见 1920×1080、2560×1440、3840×2160 等 16:9），现场通过整体缩放适配物理屏幕，保证指挥大厅、展厅与拼接屏 **一屏看全**。

## 2. 背景

曾评估 **固定宽度 + 自由高度**，希望在超高屏上减少留白。经实现验证与行业对标后 **明确放弃**，统一为与 DataV、腾讯灯塔 DataTalk 大屏、帆软大屏等一致的 **固定尺寸** 方案。

## 3. 决议依据

### 3.1 行业默认做法

主流 BI 大屏工具默认：

- 画布 **逻辑宽高固定**（如 1920×1080、3840×2160）；
- 组件 **`position: absolute` 像素级定位**；
- 展示端 **整体等比缩放** 到目标分辨率。

与用户对大屏产品的既有心智一致，降低迁移与培训成本。

### 3.2 技术：绝对定位无法可靠「撑高」

自由布局为全元素绝对定位：

- 子元素脱离文档流，父容器 **不能** 被子元素自然撑开；
- 高度只能手动指定；强行按视口或内容适配不同高度会导致 **挤压、错位、留白失控**。

因此「固定宽 + 自由高」在实现上不可作为稳定产品能力推广。

### 3.3 场景：标准大屏要一屏看全

指挥中心、展厅、拼接屏的核心诉求是 **完整信息一屏呈现**：

- 纵向滚动条在拼接屏上 **难以操作且不美观**；
- 固定逻辑画布 + 等比缩放，保证设计稿与现场观感一致。

### 3.4 产品分流：流式高度请用网格布局

若业务需要 **宽度铺满、高度随内容增长**（门户报表、可滚动分析页），应使用 **网格布局（`grid`）** 仪表盘，而非扩展自由布局的「自由高度」。

| 诉求 | 推荐布局 | 说明 |
|------|----------|------|
| 指挥大屏 / 展厅 / 拼接屏 | **自由布局** | 固定逻辑尺寸 + 等比缩放 |
| PC 门户 / 可滚动报表 | **网格布局** | 流式栅格，高度随内容 |
| 移动端单列 | **列表布局** | 固定逻辑宽度 + 列表流 |

## 4. 产品经理 / 需求评审指引

以下需求 **默认 Out of Scope** 或引导至网格布局，勿写入自由布局 Backlog：

- 「大屏高度跟着图表多少自动变长」
- 「只要宽度铺满，高度无限往下排」
- 「自由布局也要像门户一样纵向滚动看全站」
- 「拼接屏上允许整页上下滚动浏览」

若客户强制要求可滚动多屏分析，应创建 **网格布局** 仪表盘或独立专题页，而非改造自由布局画布语义。

## 5. 对白皮书与售前的意义

- **体验一致**：设计分辨率即交付分辨率，减少「上墙后布局乱了」的客诉。
- **对标清晰**：与 DataV / 帆软 / 灯塔大屏同一品类话术，便于投标与 POC。
- **边界清楚**：自由布局卖「一屏震撼」，网格布局卖「日常分析可滚动」，避免一个插件承担两种矛盾诉求。

## 6. 变更流程

若有人（含 AI）提议为自由布局恢复「自由高度」：

1. 阅读本文与 [开发者设计与实现](../develop/dashboard-position-layout.md)；
2. 说明为何网格布局不能满足场景；
3. 经产品负责人 + 前端 spec 评审后，方可修订本决议。
