# 无障碍（a11y）范围说明

> **状态**：已决议（2026-07）  
> **受众**：产品、设计、项目管理、售前  
> **关联工程说明**：[DataView 前端实现规格](https://github.com/DataLuminary/DataView/blob/main/spec/development/accessibility-scope.md)

## 1. 决议摘要

**DataLuminary 定位为 BI 与数据可视化平台，目标用户为具备正常视力的业务分析人员、管理者与数据团队。产品 **不** 将「面向盲人、弱视用户的无障碍阅读与操作优化」纳入需求范围、路线图或验收标准。**

这不是疏忽，而是与品类、场景和投入产出比一致的产品边界。

## 2. 产品定位依据

| 维度 | 说明 |
|------|------|
| 品类 | 企业 BI、经营分析、管理驾驶舱、数据大屏 |
| 核心交付 | 图表、表格、地图、指标卡等 **视觉化** 表达 |
| 典型用户 | 运营、财务、管理者、数据分析师（办公场景、鼠标 + 显示器） |
| 典型场景 | 拖拽建模、仪表盘编排、大屏展示、下钻联动、AI 配图 |

数据洞察的 **价值载体是「看」**——趋势、对比、分布、异常均需图形化呈现。即便为图表补充文字摘要，也无法替代完整、可用的非视觉分析体验，与 Tableau、帆软、DataV 等同类产品的能力边界一致。

## 3. 明确 **不** 做的范围

以下能力 **不作为** 产品需求、迭代项或对外合规承诺（除非未来经正式 spec 修订）：

- **读屏软件**（NVDA、JAWS、VoiceOver 等）下的完整可用路径
- 为复杂画布、拖拽编辑器、图表渲染层系统性补充 **ARIA / 语义化朗读**
- **纯键盘** 完成自由布局大屏编排、多选拖拽、缩放对齐等设计器操作
- 为色觉障碍用户单独设计 **全站配色体系** 或强制高对比主题（非业务主题包范畴）
- 追求 **WCAG 2.x AA** 级 Web 无障碍认证或政府无障碍采购专项适配

### 3.1 与「数据可导出」的区分

用户仍可通过 **表格导出、API、报表下载** 等渠道获取 **数据本身**；这不等同于为 **可视化编辑与阅读界面** 做读屏优化。售前与需求评审时请区分二者，勿将「数据能导出」承诺为「视障用户可独立使用全平台」。

## 4. 产品经理 / 需求评审指引

收到以下类型需求时，**默认拒绝或降级为 Out of Scope**，勿写入 `upcoming.md` 或迭代 Backlog：

- 「要支持盲人操作仪表盘」
- 「图表要能被屏幕阅读器完整朗读」
- 「所有配置项必须键盘可达且符合 WCAG」
- 「政府无障碍网站改造对标」

若客户 **强制** 要求无障碍合规，应作为 **定制交付 / 独立项目** 评估，而非主线产品能力，且须单独立项 spec 与预算。

## 5. 设计协作

- 设计稿 **不必** 为读屏顺序、焦点环、跳过导航等做专项说明。
- Ant Design 等组件库的默认交互即可；**勿** 为 a11y 单独增加交互范式或评审卡点。
- 大屏、驾驶舱以 **视觉密度与信息层次** 为优先，而非朗读顺序。

## 6. 变更流程

若有人（含 AI 助手）提议将视障 / 读屏无障碍纳入主线：

1. 阅读本文与前端 [accessibility-scope.md](https://github.com/DataLuminary/DataView/blob/main/spec/development/accessibility-scope.md)；
2. 说明目标客户、合规依据与为何导出/API 不能满足；
3. 经产品负责人 + 前端 spec 评审后，方可新开 spec **修订** 本决议。
